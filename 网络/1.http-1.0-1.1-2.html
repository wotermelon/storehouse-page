# http

# http/1.0

正式作为标准，传输内容格式不限制，增加了PUT、PATCH、HEAD、OPTIONS、DELETE 命令

# http/1.1

  • 持久连接（Keep-Alive）
  • 管道机制

# 持久连接（Keep-Alive）

http/1.0 客户端增加 Connection: Keep-Alive 头部，服务器响应返回相同的头部，没有则不持久连接 http/2.0 客户端不用增加头部，只需要服务器响应返回 Connection: Keep-Alive 头部

在没有Keep-Alive之前，http请求都是短连接，就是说每一次请求都要建立连接，请求完成后马上关闭连接。每次请求都要建立连接带来了资源的浪费，为了提高请求效率，于是有了Keep-Alice。 Keep—Alive允许在一定时间内，同一个域名多次请求数据，只建立一次http连接，其他请求可以复用这个连接通道，以达到提高请求效率的目的。

# 管道机制

浏览器要向一个域名发送多个请求，需要在本地维护一个FIFO队列，完成了一个再发送下一个，这样就存在一个问题，服务端从完成一个请求开始回传，到收到下一个请求的这段时间内是处于空闲状态的。 将浏览器的请求一股脑的打包发给服务器，服务器就可以在出开完一个请求后，马上处理下一个，不会在之前说的空闲时间。

存在的问题：

  • 队首阻塞
  • 为了保证按顺序回传，缓存多个响应，占用资源
  • 网络异常导致连接断开，无法得知服务器情况，全部重试导致重复处理
  • 中间的代理设备不一定支持http管道

# http/2.0

  • 多路复用
  • 服务端推送
  • 头部️压缩
  • 二进制帧数据传输

# 多路复用

浏览器同一域名6个通道的限制，为了解决串行传输和连接数过多，通过帧对数据进行顺序标识，一个连接并行传输数据，最后按照序列对数据进行合并。 HTTP/2对同一域名下所有请求都是基于流，同一域名不管访问多少文件，也只建立一路连接

# 服务器推送

服务端可以主动推送，客户端也有权选择是否接收。如果服务端推送的资源已经被浏览器缓存过，浏览器可以通过发送 RST_STREAM帧 来拒收

# 头部️压缩

头部压缩使用了HPACK算法，支持基于静态哈夫曼码表的哈夫曼编码

http请求都是由状态行、请求/响应头部、消息主体三部分组成。 静态字典：对头部常见键值对维护字典，减少字符。 动态字典：维护一份相同的动态字典，可以动态的添加内容。

# 二进制帧数据传输

  • 帧 http/2数据通信的最小单位。每个帧都包含帧首部，其中会标识当前帧所属的流。
  • 消息 指http/2中逻辑上的http消息。例如请求和响应等，消息由一个或多个帧组成。
  • 流 存在于连接中的虚拟通道。流可以承接双向消息，每个流都有一个唯一的整数id，HTTP/2对同一域名下所有请求都是基于流。

http/2将请求和响应数据分隔成为更小的帧，并对他们采用二进制编码。 同域名下的所有请求都在一个连接上完成，这个连接可以承载任意数量的双向数据流。 每个数据流都以消息的形式发送，消息由一个或多个帧组成。多个帧之间可以乱序发送，然后根据帧首部的流标识可以重新组装。
